# Language Zones and Build Targets

Doc-Class: canon
Status: active
Date: 2026-07-06
Owner: Firegrid Architecture
Substrate: S2

Decision record for the F#/Fable commitment. Ends the split-brain state between
the F# `src/` tree and the Effect-TS `packages/` tree by assigning every module
a home by rule, not by history.

## The Two-Zone Rule

**F# owns everything that writes S2.** The substrate client, folds,
checkpoint/trim, turn logs, lifecycle authority, wake routing, timers — the
durable core. Platform capability work (managed-sessions lanes A/B/C) is F#.

**TypeScript owns everything that touches the TS ecosystem.** The proof
harness (`apps/proofs`), harness adapters (Claude Agent SDK, ACP — their SDKs
are TS-native), the protocol client SDK, and consumers such as agent-ui.
Adapter and consumer work (lanes D/E) is TS.

A module that seems to need both zones is a design smell: split it at the
protocol or the seam below.

## The One-Seam Law

Exactly two ways to cross between zones:

1. **The kernel protocol** — ingress/attach/control surfaces, per the
   runtime-fronted posture in [`architecture.md`](./architecture.md). Processes
   (adapters, consumers, the proof harness) prefer this edge.
2. **One Fable-emitted package seam** — the Fable build of the kernel,
   published under the existing substrate package names. Emit TypeScript
   (`fable --lang typescript`) so the seam's types are generated from the F#
   source and cannot drift; wrap with a thin hand-written idiomatic TS facade
   for npm consumers (emitted TS is structurally F#-shaped and is not the
   public API). Library-level access (custom TS hosts, law-level proofs) uses
   this edge only. API design inside the F# zone follows
   [`../../sdds/fsharp-fable-effsharp-evaluation-sdd.md`](../../../sdds/fsharp-fable-effsharp-evaluation-sdd.md).

No other deep imports across zones. TS proofs driving an F# kernel are
structurally black-box, which mechanically enforces the proofs-consume-only-
public-exports rule in `AGENTS.md`.

## P3 Target Surface: `Foundation/Durable` Kernel

Status: G6/G1 approved 2026-07-07 for implementation. The Processor signatures
below are the cross-lane G1 shape; B1, C1, and A-lane work build against them
as-is, and any implementation-forced change comes back through architect
review.

P3 ports the proven eff-firegrid durable kernel as the F#-native seed for the
actor composition in
[`authority-and-actors.md`](./authority-and-actors.md). The port is
EffSharp-free and keeps the sans-IO split explicit: replay semantics,
step-planning, codecs, mailbox admission selection, and handler decisions are
pure; S2 reads/writes, fence ownership, activity/timer dispatch, and timestamp
sampling live in shells.

P3 is a port/refactor work packet. It does not deliver B1's `Authority`
protocol (I5), the generic `DurableLog`/Turn binding (I1), checkpoint records
(I4), wake records (I3), or any agent-ui integration. Where the donor code has
`InboxFold`, P3 generalizes the public module name to `Mailbox` but preserves
the proven source-sequence/highwater semantics.

### Pure Replay Surface

The durable authoring core is a small, replayable program representation plus a
builder. The program type remains opaque enough that callers construct values
through `Durable`/`Workflow` helpers rather than by depending on union cases
unless a proof explicitly targets replay semantics.

P3 deliberately preserves the donor kernel's positional `OpId` replay model:
operation ids are assigned by durable-program position, including source-order
fan-out. This is sound for the sequential `durable {}` CE port and keeps P3 in
the "ports are refactors" lane. It is a recorded deviation from the keyed
replay direction in the canon lineage: code evolution can invalidate in-flight
histories. Re-open this decision when the authoring surface becomes product or
when cross-implementation history compatibility is claimed.

```fsharp
namespace Firegrid.Store.Foundation.Durable

type OpId = OpId of int

type Payload = string

    | Timer of deadline: int64

    | RaceActivity of Activity
    | RaceEvent of EventKey

    | ActivityWon of index: int * value: Payload

type Event =

    | TimerCanceled of OpId

type History

    | NeedsActivity of Activity

    | Done of 'a

    val perform : Activity -> Durable<Payload>
    val performAll : Activity list -> Durable<Payload list>
    val await : EventKey -> Durable<Payload>
    val whenAny : RaceTask list -> Durable<RaceResult>

```

### Step Records, Codecs, And Planner

The step planner converts replay `Need`s into durable records. Codecs use
length-prefixed text encodings with golden-vector tests so future host targets
can read the same S2 streams.

```fsharp
type StepCommand =
    | CallActivity of OpId * Activity
    | ScheduleTimer of OpId * deadline: int64
    | CancelTimer of OpId
    | WriteLog of OpId * message: string

type MailboxMessage =
    | StartWorkflow of WorkflowName * Payload
    | RaiseSignal of name: string * payload: Payload
    | CompleteActivity of OpId * Payload
    | FireTimer of OpId * deadline: int64

type MailboxEnvelope =
    { Source: string
      SourceSeqNum: int64
      Message: MailboxMessage }

type StepRecord =
    | WorkflowStarted of WorkflowName * Payload
    | HistoryEvent of Event
    | Command of StepCommand
    | SignalDelivered of source: string * sourceSeqNum: int64 * opId: OpId
    | CommandDispatchCheckpoint of dispatcher: string * nextSeqNum: int64
    | MailboxCheckpoint of nextSeqNum: int64
    | MailboxSourceHighwater of source: string * nextSeqNum: int64
    | MailboxMessageAccepted of MailboxEnvelope

type StepPlan<'a> =
    | Complete of 'a
    | Commit of HistoryEntry<StepRecord> list
    | Waiting of OpId * Need

type StepCommitResult =
    | StepCommitted of S2.AppendAck
    | StepNotRequired
    | StepDeposed of expectedFence: string
    | StepCommitFailed of S2Errors.S2Failure

module MailboxEnvelopeCodec =
    val encode : MailboxEnvelope -> string
    val decode : string -> Result<MailboxEnvelope, string>

module StepRecordCodec =
    val encode : HistoryEntry<StepRecord> -> string
    val decode : string -> Result<HistoryEntry<StepRecord>, string>

module Stepper =
    val historyFromRecords : HistoryEntry<StepRecord> list -> History
    val plan : timestamp:int64 -> History -> Durable<'a> -> StepPlan<'a>
    val commit :
        encode:(HistoryEntry<StepRecord> -> string) ->
        owned:OwnedLog ->
        StepPlan<'a> ->
        Async<StepCommitResult>
```

`WorkflowName`, `HistoryEntry<'a>`, `OwnedLog`, and the S2 error/ack types are
the public F# substrate types already owned by `Firegrid.Store` and
`Firegrid.Log`.

### Mailbox Admission

Mailbox admission is the generalized `InboxFold` port. It reads the actor
mailbox from the last committed mailbox checkpoint, dedupes by
`(Source, SourceSeqNum)`, translates admitted messages into step records, and
commits progress under the current fence.

```fsharp
type MailboxAdmittedMessage =
    { MailboxSeqNum: int64
      Envelope: MailboxEnvelope
      Events: Event list }

type MailboxReport =
    { FromSeqNum: int64
      NextSeqNum: int64
      Scanned: int
      Accepted: MailboxAdmittedMessage list
      Duplicates: int
      Commit: CommitResult option }

type MailboxFailure =
    | LogReadFailed of string
    | LogDecodeFailed of seqNum: int64 * error: string
    | MailboxReadFailed of string
    | MailboxDecodeFailed of seqNum: int64 * error: string
    | CommitFailed of S2Errors.S2Failure

type MailboxStatus =
    | Admitted of MailboxReport
    | Deposed of expectedFence: string
    | Failed of MailboxFailure

module Mailbox =
    val runOnce :
        encodeLog:(HistoryEntry<StepRecord> -> string) ->
        decodeLog:(string -> Result<HistoryEntry<StepRecord>, string>) ->
        decodeEnvelope:(string -> Result<MailboxEnvelope, string>) ->
        maxRecords:int ->
        owned:OwnedLog ->
        Async<MailboxStatus>
```

### Processor Shell

The processor is the generic actor shell named by `authority-and-actors.md`.
P3 ports the proven host loop mechanics and expresses the pure handler boundary
the later B/A/C lanes bind to. It does not introduce a separate actor
framework, resident process model, or actor-to-actor call primitive.

Provenance is stamped by the shell, not by handler code. A handler may request
`Intent.Send(target, message)`; the shell records the intent, derives the
outbound mailbox `Source` and `SourceSeqNum` from that committed intent
record's identity, and then appends the stamped envelope to the target mailbox.
Handlers never fabricate mailbox provenance.

```fsharp
type ActorAddress = { Segments: string list }
type TimerId = TimerId of string
type Timestamp = int64
type EffectId = EffectId of string
type SubjectId = SubjectId of string

type WakeReason =
    | MailboxReady
    | TimerFired of TimerId * Timestamp
    | ChildTerminal of SubjectId

type Intent =
    | SetTimer of TimerId * dueAt: Timestamp
    | Send of target: ActorAddress * message: MailboxMessage
    | Execute of EffectId * payload: Payload

type StoredRecord<'record> =
    { Seq: int64
      Body: 'record
      Timestamp: Timestamp }

type Admitted<'msg> =
    { MailboxSeqNum: int64
      Source: string
      SourceSeqNum: int64
      Message: 'msg }

type Decision<'state, 'record, 'terminal> =
    { State: 'state
      Append: 'record list
      Intents: Intent list
      Seal: 'terminal option }

type Handler<'state, 'msg, 'record, 'terminal> =
    { Initial: 'state
      Fold: 'state -> StoredRecord<'record> -> 'state
      OnAdmitted: 'state -> Admitted<'msg> -> Decision<'state, 'record, 'terminal>
      OnWake: 'state -> WakeReason -> Decision<'state, 'record, 'terminal> }

type DriveOutcome<'terminal> =
    | Idle
    | Advanced
    | Sealed of 'terminal
    | Deposed of expectedFence: string
    | Failed of DriveError

and DriveError =
    | ClaimFailed of string
    | RebuildFailed of string
    | MailboxFailed of MailboxFailure
    | CommitFailed of S2Errors.S2Failure
    | IntentDispatchFailed of string
    | Unexpected of string

type DriveEnv

module Processor =
    val drive :
        DriveEnv ->
        ActorAddress ->
        Handler<'state, 'msg, 'record, 'terminal> ->
        Async<DriveOutcome<'terminal>>
```

`DriveEnv` is the shell dependency record: clock/timestamp source, S2 stream
access, fence ownership, mailbox codecs, log codecs, and idempotent intent
dispatch. Its fields are implementation-level until B1/A/C lanes bind them to
their approved interfaces.

### Runtime And Test Host

The donor durable runtime is retained as a proof/dev harness around the kernel,
not as the future product API for managed sessions. It is useful for proving
the ported CE, mailbox admission, activity/timer dispatch, and host tick laws.
`Processor.drive` is the only production drive loop; any future workflow
binding is a `Handler` over it, never a second coordination loop.


is allowed in the semantic core; the runtime shell may provide defaults only at
the executable/test edge.

### Laws And Proof Obligations

- **Replay determinism:** given the same `History`, `Durable.replay` produces
  the same `Outcome`; replaying a completed history is effectively-once; every
  strict prefix blocks rather than inventing a result.
- **Step id stability:** operation ids are positional and stable under replay;
  fan-out preserves source order even when completions arrive out of order.
- **Intent-before-effect:** the planner commits command records before
  activity/timer/log side effects are dispatched. A crash after commit and
  before dispatch causes idempotent redispatch, not a second decision.
- **Mailbox dedupe:** mailbox admission folds each `(Source, SourceSeqNum)` at
  most once, persists source highwater, and resumes from the latest mailbox
  checkpoint after restart.
- **Fails closed:** malformed log or mailbox records fail the tick without
  committing partial progress.
- **Fence safety:** every log progress commit happens through the owned/fenced
  shell; a stale holder returns `Deposed` and cannot commit mailbox progress,
  step records, signal delivery, activity completion, timer publication, or
  terminal state.
- **Sans-IO split:** replay semantics, step planning, codecs, mailbox
  selection, and handler decisions are pure. S2, activity execution, timer
  execution, timestamp sampling, id generation, and trace emission stay in
  shells.
- **Public proof surface:** P3 proofs import only `Firegrid.Store.Foundation`
  durable modules and existing public proof/runtime helpers. They must not
  deep-import private generated files or add proof-only branches to production
  code.

## P4 Target Surface: `Firegrid.Log` TypeScript Facade

Status: G6 surface sketch for architect review. Because this is the one package
seam consumed across zones, its shape is also a G1 interface decision. Do not
implement or re-point proofs until this section is approved.

P4 supplies the package seam for `Firegrid.Log`: generated TypeScript from the
F# source is the typed truth, and a thin hand-written TypeScript facade is the
public npm API. The facade is a product surface for TypeScript hosts and law
proofs, not a proof-only adapter.

### Placement

| Layer | Location | Committed? | Role |
| --- | --- | --- | --- |
| F# source | `src/Firegrid.Log/S2/*.fs` | yes | Owns S2 client behavior, normalized records, errors, and Fable-exported operation types. |
| Fable TypeScript output | `src/Firegrid.Log/dist-ts/**` or equivalent build output | no | Generated typed truth for the seam; regenerated by package build. |
| TS facade source | `src/Firegrid.Log/ts/**` | yes | Thin Promise-first TypeScript API over generated Fable output. |
| Effect wrapper source | `src/Firegrid.Log/ts/effect.ts` | yes | Optional Effect idiom for proof/infrastructure consumers. |
| npm export | `@firegrid/log` | yes | Promise-first public facade only; generated Fable modules are not documented public subpaths. |
| npm export | `@firegrid/log/effect` | yes | Thin wrapper over the public facade for Effect consumers. |

The package export contract after P4 is:

```json
{
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "default": "./dist/index.js"
    },
    "./effect": {
      "types": "./dist/effect.d.ts",
      "default": "./dist/effect.js"
    }
  }
}
```

If runtime packaging requires generated Fable files under `dist/`, they remain
implementation files imported by the facade. Product consumers import
`@firegrid/log`; proof and infrastructure code may import `@firegrid/log/effect`
when Effect composition is useful.

subclasses, Promises, and async iterables. This matches the Fable-emitted
Promise-shaped exports and the F# API doctrine. The first production consumer
(`agent-ui`) and future TypeScript hosts are Promise-native; wrapping Promises
in Effect is trivial, while forcing Promise consumers through Effect embeds a
runtime at the boundary.

### Laws

- **Typed truth:** every public facade type that corresponds to an F# type or
  operation is mechanically backed by the Fable TypeScript output. The
  hand-written facade may rename, normalize casing, and wrap promises, but it
  must not synthesize behavior absent from the F# source.
- **Single import:** product consumers import the facade from `@firegrid/log`;
  Effect consumers may import `@firegrid/log/effect`; no public or proof code
  imports generated Fable paths.
- **Error identity:** conditional append and range failures are surfaced as
  `instanceof`-checkable subclasses (`SeqNumMismatchError`,
  `FencingTokenMismatchError`, `RangeNotSatisfiableError`) with the recovery
  fields used by callers. If the current generated F# output lacks a recovery
  field, implementation updates the F# export surface before wrapping it.
- **Effect boundary:** the root facade's data records, constructors, errors,
  operations, and streams stay Effect-free. Effect appears only in
  `@firegrid/log/effect`, and that submodule wraps the Promise-first root
  facade without adding behavior.
- **Record fidelity:** `AppendInput.create` and `AppendRecord.*` lower
  one-to-one to the generated F# record/command constructors, including fence
  and trim command records. The facade does not reinterpret command records.
- **Read fidelity:** root `readSession` preserves stream order and
  command-record filtering semantics from the F# client as an `AsyncIterable`.
  The Effect wrapper exposes the same session as a `Stream` without buffering
  beyond what is needed to bridge the runtime. Effect interruption of the
  wrapper stream closes the underlying S2 read session so long-running readers
  do not leak substrate sessions.
- **Differential evidence:** after implementation, the active substrate proofs
  gain `firegrid-log` copies that drive this facade, while the existing
  `upstream-sdk` oracle copies remain. The conformance bridge only marks
  Firegrid client conformance for the `firegrid-log` driver.
- **No second transport:** the facade never constructs `@s2-dev/streamstore`
  directly. Upstream SDK use stays inside the F# S2 client or in the retained
  oracle proofs.

## EffSharp Is Deprecated

Ruled 2026-07-06. EffSharp was a bridge library built while the mental model
was still TS-Effect; the F# API doctrine's own conclusion — F# natively covers
most of what Effect gives TypeScript — applies to the bridge itself. It is also
the sole reason for the private GitHub Packages NuGet feed and its recurring
local-auth friction.

- **No new EffSharp.** New F# code uses native shapes: `Result` + DUs for
  errors, `Async`/`task`/`promise` CEs at shells, records/parameters for
  dependencies, and the pull-cursor pattern (`readCursor`/`tryNext`/`close`)
  for stream consumption — the shape the eff-firegrid foundational design
  already proved without EffSharp.
- **Removal is WP P5**: strip EffSharp from `Firegrid.Log`, `Firegrid.Store`,
  and `Firegrid.Foundation.Proofs`, then delete the private NuGet feed from
  `nuget.config`. The CI-green proof suites gate the refactor.
- **P3 lands EffSharp-free from the start** (the eff-firegrid durable kernel is
  `Async`-based already).
- **Distinct from Effect (TS).** EffSharp (F#-side Effect port) and Effect
  (TS library) are different dependencies. Effect's justified TS footprint is
  narrow: it is the proof harness's internal idiom (structured concurrency,
  fault injection, scoped cleanup, OTel weave — verification infrastructure
  that never ships to consumers) plus optional thin wrapper modules. It is not
  the default idiom for public TS surfaces.

The layering at the Fable boundary: generated exports are Promise-shaped
(doctrine rule); the public facade is **Promise-first** (plain types, `Error`
subclasses, async iterables) because the first production consumer (agent-ui)
and future TS hosts are Promise-native; Effect consumers get a thin wrapper
submodule (e.g. `@firegrid/log/effect`). Wrapping promises in Effect is
trivial; the reverse requires embedding a runtime — which is why Promise-first
is the only ordering where the secondary idiom stays thin. Nothing at the seam
requires an effect system on either side.

## The Sans-IO Core Rule

Kernel *semantics* are pure F# modules: `fold : State -> Record -> State`,
`plan : State -> Intent list`, codecs, version math, predicate evaluation.
No async, no HTTP, no ambient clock or randomness — time and entropy enter as
data. I/O lives in per-target shells.

This is what makes the multi-target story cheap:

| Target | Core | Shell | Status |
| --- | --- | --- | --- |
| JS / Node | Fable→JS | Node S2 client; kernel host + TS bindings | Active (this repo) |
| Rust data plane | Fable→Rust core crate | Hand-written host over the native `s2-sdk` crate (seed: eff-firegrid PR #63 `firegrid-host`) | Deferred until a perf requirement writes its SDD |
| dotnet | Native | FsCheck property tests / fuzzing of pure semantics | Opportunistic |

Fable's Rust backend handles the pure-functional F# subset and little more —
the sans-IO rule keeps the core inside that subset by construction. Do not open
a Rust lane ahead of a demanding consumer; the option is kept warm by this rule
plus golden vectors, at near-zero cost.

## Cross-Target Conformance

- **Golden vectors** for every codec (step records, turn chunks, checkpoint
  records): byte-level fixtures checked by all targets, so differently-built
  hosts can serve the same S2 streams in a mixed fleet.
- **The TS proof suite** drives any host build through the same public
  protocol; proofs are target-blind, so the conformance suite gates every
  plane for free.

## Consequences for Agents

- Target Surfaces (gate G6) for F#-zone lanes are F# signatures (modules,
  DU-typed errors); TS-zone lanes keep Effect shapes per `LLMS.md`.
- The known risk is agent fluency in F#: mitigate with the ported eff-firegrid
  code as the exemplar corpus and strict G6 review. The F# compiler is the
  stricter reviewer; lean on it.
